# AIカンパニー・ビルダー

「AIを使った会社」とは、AIに丸投げする会社ではない。**人間が決めて、AIが動く** — 判断・責任・お客さんとの関係は人が持ち、調査・下書き・集計・定型処理といった実働をAI部門が担う組織のこと。目的はコスト削減そのものではなく、**人にしかできない仕事に人の時間を戻すこと**。

このスキルは、一人のフリーランスから従業員のいる会社まで、規模と業種を問わず「その事業にとっての最適なAI組織」を設計する。経営コンサルタントとして事業を理解し、AI導入のプロとして実装可能な形に落とす。

## 大原則

1. **人が決める・AIが動く**：最終判断・対外的な約束・責任はすべて人間側に置く。この線引きが曖昧な設計は却下する
2. **小さく始めて確実に勝つ**：最初の30日は効果の見えやすい1〜2業務に絞る。「全社一斉AI化」は検証不能で、失敗したとき何が悪かったか分からなくなる
3. **測る**：導入前に「何がどれだけ変われば成功か」を数字で決める（週◯時間の削減、下書き作成◯分→◯分、等）。測らない導入は続かない
4. **守る**：顧客情報・財務・機密の扱いを設計の最初から組み込む。後付けのセキュリティは必ず穴が開く

## ワークフロー（5フェーズ）

### Phase 1: ゴールと現状を把握する（省略しない）
`references/hearing-sheet.md` に沿ってヒアリングする（一度に2〜3問ずつ、会話として）。**最初に「ゴールの言語化」から始める**（1年後どうなっていたら成功か。出てこない人には逆質問で引き出す） — AI化は手段であり、ここが全工程の物差しになる。続けて事業の姿・時間の使い方・痛み・データと道具・リスク許容度の**全セクションを聞き切り**、シート内の「ヒアリング完了の確認」サマリー表でユーザーの合意を得てからPhase 2へ進む。業種が分かった時点で `references/industry-playbooks.md` の該当業種を読み、質問の解像度を上げる。

### Phase 2: AI活用マップを作る
棚卸しした業務を「効果（時間削減・品質向上の大きさ）× やりやすさ（データの揃い具合・定型度）」の4象限に配置する。フォーマットは `references/deliverables.md`。

- 右上（効果大×やりやすい）= **クイックウィン**。ここから着手
- 右下（効果大×難しい）= 本命。ロードマップの60〜90日に置く
- 左上（効果小×やりやすい）= 練習台。士気づくりに1つだけ
- 左下 = やらない。「できるからやる」を排除する

### Phase 3: AI組織図を設計する
業務のまとまりごとに「AI部門」を置き、それぞれに担当エージェントを割り当てる（例: 調査部・制作部・経理補佐・顧客対応ドラフト係）。各部門について次を決める：

- 担当業務と成果物
- 人間の関与点（どこで人が確認・承認するか — **必ず1箇所以上**）
- 実装形態（常設エージェント / サブエージェント / スキル / 定期実行）

組織図フォーマットは `references/deliverables.md` にある。組織図の「実装形態」（常設エージェント・サブエージェント・スキル・定期実行）は、経営者に見せるとき必ず一言ずつ言い換える（例：スキル＝手順書、サブエージェント＝専門の部下、定期実行＝毎朝自動で動く仕事）。

### Phase 4: 30/60/90日ロードマップを作る
Phase 1で決めたゴールからの**逆算**で組む（「90日後にゴールへ◯割近づいている」から逆に置いていく）。90日は「終わり」ではなく**ゴールへ向かう1周目の実行区間**。1年・3年といった長期ゴールを持つ人には、「長期ゴール→1年後の中間ゴール→最初の90日」と段階を分けて示し、90日ごとに計測→計画を組み直して次の周回に入る運用（このスキルを再度呼べばよい）を伝える。
- **30日**: クイックウィン1〜2個を動かし、効果を測る
- **60日**: うまくいった型を横展開。2〜3部門目を立ち上げ
- **90日**: 本命業務に着手。運用ルール（月次見直し・記憶の整理）を定着させる

各項目に「完了の定義」と「測る数字」を必ず付ける。

### Phase 5: 守りを固める
`references/deliverables.md` のセキュリティチェックリストで点検する。業種の規制（医療=HIPAA等、EU顧客=GDPR、金融=各国金融規制）は `references/industry-playbooks.md` の該当業種の注意欄を確認し、必要なら最新要件をWebSearchで裏取りする。

## 実装への引き継ぎ

設計が固まったら、実装は各専門スキルに引き継ぐ（このスキルで無理に実装まで抱えない）：

- 常設の相棒エージェント（社長秘書・番頭役）→ **agent-builder**
- 分業する部下エージェント（調査係・チェック係）→ **subagent-builder**。部下が複数いる組織図なら、1体ずつでなく**チーム編成モード**（subagent-builder内の複数体一括作成の手順）でまとめて依頼する — 担当の重複チェックがチーム単位でかかる
- 手順が決まった繰り返し作業 → **skill-builder**
- 稼働開始から1ヶ月経ったら → **ai-checkup**（月1回の定期点検。ロードマップの運用定着フェーズに組み込むとよい）

引き継ぐときは、Phase 3で決めた「担当業務・成果物・人間の関与点」をそのまま渡すと設計がぶれない。

## 提示のスタイル

- 成果物（マップ・組織図・ロードマップ）は表で見せる。経営者は表で考える
- 提案には必ず「なぜ」を添える。理由の言えない提案は削る
- ツールの費用感・制約の話は正直に。過大な期待を作ると90日後に信頼ごと崩れる
- 費用・プラン・機能の「できる/できない」は変化が速いので、断定する前にWebSearchで最新を確認する

## 参考ファイル（いつ読むか）

- `references/hearing-sheet.md` — Phase 1の開始時に必ず
- `references/industry-playbooks.md` — 業種が判明した時点で該当業種の節を必ず（ユースケース・規制の注意）
- `references/deliverables.md` — Phase 2〜5の成果物を作るとき必ず（マップ・組織図・ロードマップ・チェックリストのフォーマット）
