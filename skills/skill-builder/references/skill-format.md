# スキルファイルの書式（2026-07時点）

## 基本構造

保存場所: `~/.claude/skills/<スキル名>/SKILL.md`（全プロジェクト共通）またはプロジェクトの `.claude/skills/<スキル名>/SKILL.md`（その場所専用）。

```markdown
---
name: monthly-invoice
description: 毎月の請求書を作るスキル。「請求書作って」「今月の請求」「◯◯さんへの請求書」のように言われたら必ず使うこと。ヒアリング済みの型・敬語・振込先の書き方で仕上げる。
---

# 月次請求書の作成

## 手順
1. （手順を理由つきで）
2. …

## 出力の型
（完成品のフォーマットをそのまま示す）

## 判断ルール
- ◯◯の場合は△△（理由）

## やってはいけないこと
- （過去の失敗から）
```

frontmatterの必須は `name`（小文字とハイフン）と `description` のみ。シンプルが正義。

## descriptionの書き方（発動率はここで決まる）

スキルはdescriptionを見て発動が判断される。「何をするか」だけでは足りず、**どんな言葉で頼まれたら使うか**まで書く。

- ❌ 悪い例: 「請求書を作成するスキル」
- ⭕ 良い例: 「毎月の請求書を作るスキル。『請求書作って』『今月の請求』『◯◯さんへの請求書』のように言われたら必ず使うこと」

ユーザーが実際に言いそうなフレーズを2〜4個入れる。「〜のときは必ず使うこと」という一押しも発動率を上げる。逆に、隣のスキルと同じフレーズを書くと取り合いになるので、既存スキルのdescriptionと突き合わせて棲み分ける。

## 参照ファイルへの分割の目安

- SKILL.md本文は理想500行以内。毎回全部読み込まれるので、長いほどコストがかかり精度も鈍る
- 長い見本・資料・一覧（過去の完成品、価格表、文例集）は `references/` フォルダに分け、本文から「◯◯のときに `references/xxx.md` を読む」と指示する。こうすると必要なときだけ読まれる
- 見本を置くときは「これは完成品の実例。型と言い回しの手本にする」と用途を書き添える

## ありがちな失敗と対策

| 失敗 | 対策 |
|---|---|
| 作ったのに発動しない | descriptionに実際の頼み方フレーズを追加 |
| 例外的な依頼で壊れる | 手順に理由を添える（理由があればAIが応用する） |
| だんだん現実とズレる | 「毎回直してる箇所」に気づいたら追記。月1で見直し |
| 本文が肥大化して遅い | 見本・資料をreferences/へ分割 |
