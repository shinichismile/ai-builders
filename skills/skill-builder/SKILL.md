# スキル・ビルダー

スキルとは、AIに渡す「**あなた専用のレシピカード**」。一度作れば、次から「請求書作って」の一言で、あなたの手順・あなたの型・あなたの注意点どおりに仕事が仕上がる。同じ説明を毎回繰り返す時間が消える。

エンジニアでなくても作れる。必要なのはAIの知識ではなく、「その作業を口で説明できること」だけ。説明の聞き取りと、ファイルへの変換はこのスキルがやる。

## スキル化に向く仕事・向かない仕事

- **向く**：月3回以上やる × 手順を言葉で説明できる（請求書、週報、議事録の型、商品説明文、発注前チェック、返信メールの型…）
- **向かない**：毎回判断が大きく変わる仕事（経営判断、新規企画）→ それはスキルでなくエージェントの役割定義（agent-builder）の領分。また、一度きりの作業はスキル化するだけ無駄なのでその場でやる

判断に迷ったら「同じ説明を過去に2回以上した仕事か？」で決める。

## ワークフロー

### 1. ヒアリングする（省略しない）
以下を**すべて**確認する（一度に2〜3問ずつ、会話として。すでに分かっていることは聞き直さない）：

- **作業の流れ**：最初から最後まで、普段どおりの手順を話してもらう。「まず何をしますか？その次は？」と実演のように辿る
- **完成品の見本**：過去の実物（作った請求書・書いた週報等）があれば見せてもらう。**見本1つは説明10分に勝る** — 型・言い回し・体裁はここから抽出する
- **判断が入る箇所**：「場合によって変わるところ」はどこか。その分かれ方のルール
- **合格ライン**：何ができていれば合格か。よくある失敗・past自分がやらかしたミス
- **起動の合図**：どんな言葉で頼みたいか（→description設計の材料）

聞き終えたら要約を見せて「この理解で合っていますか」と確認してから次へ。

### 2. 設計して合意を取る
次を1枚で提示する：スキル名 / 起動の合図（description案）/ 手順の骨子 / 出力の型 / 参照ファイルの有無（見本や資料が長い場合は分割する）。

### 3. 生成する
`references/skill-format.md` の仕様に従ってSKILL.md（必要なら参照ファイルも）を書き、`~/.claude/skills/<スキル名>/` に保存する。書くときの原則：

- **descriptionには「何をするか」＋「どんな言葉で頼まれたら使うか」の両方**を入れる。ここが曖昧だと、作ったのに発動しないスキルになる（一番多い失敗）
- 手順は**なぜそうするかの理由つき**で書く。理由があるとAIは応用が利く。丸暗記の指示は例外が来た瞬間に壊れる
- 本文は簡潔に（長くても500行以内）。長い見本・資料は`references/`に分けて「◯◯のときに読む」と指示する

### 4. テストする
その場で1回、実際の作業を頼んでもらって動かす。見るのは「ちゃんと発動したか」「完成品が合格ラインを超えたか」の2点。ズレたら、多くの場合directionか手順の理由不足なので直す。

### 5. 育て方を伝える
使っていて「ここ毎回直してるな」と思ったら、その修正をスキルに追記してもらう（このスキルを再度呼んで「◯◯スキルを改善したい」でもよい）。スキルは作った日が最低品質で、使うほど良くなる。

## 品質チェックリスト（生成後に必ず自己点検）

- [ ] descriptionだけ読んで、いつ発動すべきか第三者に分かるか
- [ ] 手順に理由が添えてあるか（丸暗記指示になっていないか）
- [ ] 完成品の型（出力フォーマット）が具体的に書いてあるか
- [ ] 判断が分かれる箇所の分かれ方ルールが書いてあるか
- [ ] 機密情報（顧客名・価格の内部ルール等）を入れる場合、このスキルを人に渡さない前提か確認したか

## 参考ファイル（いつ読むか）
