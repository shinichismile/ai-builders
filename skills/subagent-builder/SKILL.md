# サブエージェント・ビルダー

サブエージェントは「自分専用のコンテキスト・ツール・権限を持つ専門ワーカー」。親の会話を汚さずに作業し、要点だけを持ち帰る。このスキルは、どんな職種のユーザーでも（エンジニアでなくても）最高品質のサブエージェントを作れるよう、ヒアリングから設計・生成・テストまでを一気通貫で行う。

初心者に説明するときは「新しい部下を1人採用する」という例えを使うとよい。名前（name）、仕事内容（description＝任せる基準）、持たせる道具（tools）、能力とコスト（model）、業務マニュアル（本文＝システムプロンプト）を決めるのが採用面接、という構図。

## 設計の5原則

1. **単一責任**：1体のサブエージェントには1種類の仕事だけを持たせる。「何でもやる部下」は結局どの場面でも委任されず、指示も曖昧になる。仕事が2つあるなら2体作る。
2. **descriptionが命**：親のClaudeは`description`だけを見て「この仕事をこの部下に任せるか」を判断する。「〜の専門家」だけでは足りない。**どんな場面で・どんな依頼が来たら使うのか**を具体的に書く（「コード変更後は積極的に使うこと」のような一押しも有効）。
3. **最小権限**：読むだけでいい仕事にWrite/Editを持たせない。事故を防ぐだけでなく、余計な行動の選択肢が消えることで出力の質も上がる。`tools`（許可リスト）か`disallowedTools`（拒否リスト）で絞る。
4. **モデルはコストと難易度で選ぶ**：定型的な調査・分類はhaiku、標準的な作業はsonnet、高度な判断はopus以上。サブエージェントは回数が積み重なるので、モデル選択がそのままコスト差になる。
5. **本文＝システムプロンプトのすべて**：サブエージェントにはClaude Code本体のシステムプロンプトは届かない。本文に書いたことだけが業務マニュアルになる。役割・手順・出力形式・持ち帰る報告の形（「要点だけ・◯行以内」等）まで本文で定義する。

## ワークフロー

### 1. ヒアリングする（省略しない）
いきなり作らず、以下を**すべて**確認する（一度に2〜3問ずつ、会話として。会話の中ですでに答えが出ている項目だけは聞き直さない。専門用語が通じない相手には例えで聞く）：

- **職種**：どんな部下が欲しいか（調査係・チェック係・下書き係・実装係…）。イメージが曖昧なら「今、誰かに任せられたら一番楽になる仕事は？」から掘る
- **仕事内容**：何をやらせたいか。入力は何で、成果物は何か。良い成果物の条件（合格ライン）
- **頻度と場面**：どんなときに呼ばれるか（→descriptionの材料）
- **判断基準とNG**：迷ったときどう判断してほしいか。やってはいけないことは何か（→本文の材料）
- **危険度**：ファイルを書き換えるか、外部に何かを送るか（→ツール権限の材料）
- **求める質とコスト感**：速さ重視か、精度重視か（→モデル選択の材料)
- **使う場所**：このプロジェクトだけか、全プロジェクト共通か（→保存場所の材料）

聞き終えたら、答えの要約を一覧で見せて「この理解で合っていますか」と確認してから設計に進む。空欄のまま設計に入らない。

### 2. 設計図を提示して合意を取る
ファイルを書く前に、次の表で設計を見せて確認する：

| 項目 | 設計 | 理由 |
|---|---|---|
| name | 例: contract-reviewer | 小文字とハイフンのみ |
| description | （委任の判断基準になる文） | なぜこの場面指定か |
| tools | 例: Read, Grep, Glob | なぜこの絞り方か |
| model | 例: sonnet | なぜこのコスト帯か |
| その他 | memory / background / isolation 等 | 使う場合のみ |

### 3. ファイルを生成する
`references/frontmatter-spec.md` の正確な仕様に従って書く。似た用途の完成例が `references/patterns.md` にあるので、必ず近いパターンを確認してから書く（ゼロから発明しない）。

### 4. 保存して有効化する
- このプロジェクト専用 → `.claude/agents/`（チーム共有ならgit管理に含める）
- 全プロジェクト共通 → `~/.claude/agents/`
- **注意**：ファイルを直接置いた場合はセッション再起動で読み込まれる。すぐ試したい場合はその旨を伝える。

### 5. テストして改善する
実際の仕事を1つ任せてみて、(a) ちゃんと委任されたか（されなければdescription不足）、(b) 成果物の質、(c) 持ち帰る報告が簡潔か、を確認する。直すのは多くの場合descriptionか本文の出力指示。

## 複数体をまとめて作る（チーム編成モード）

「部下を何人か作りたい」「調査係と経理係とチェック係が欲しい」のように複数体の依頼が来たら、1体ずつバラバラに作らず、**先にチーム全体を設計してから一括生成する**。手順：

1. **仕事の全体像をヒアリング**する（上記の項目を、個別でなく業務全体について聞く）
2. **分業案を提示する**。チーム表で見せて合意を取る：

| # | name | 職種 | 担当（単一責任） | tools | model |
|---|---|---|---|---|---|
| 1 | research-assistant | 調査係 | 市場・競合の調査報告 | Read, WebSearch, WebFetch | haiku |
| 2 | invoice-checker | 経理係 | 請求書・仕訳の点検 | Read, Glob, Grep | sonnet |

3. **担当の重複チェック**を必ずやる。複数体で一番多い事故は「仕事の境界が曖昧で、親がどっちに任せるか迷う」こと。各体のdescriptionを並べて読み、同じ依頼文が2体にマッチしないか確認する。曖昧な境界があれば、descriptionに「◯◯の場合は△△（別の部下の名前）に任せる」と棲み分けを書く
4. **一括生成**する。1体ずつ全ファイルを書き、それぞれ品質チェックリストを通す
5. **テストは1体ずつ**。全部を一度に試すと、うまくいかないとき原因が切り分けられない

**作れる体数に上限はない。** 10体でも20体でも、ユーザーが望む数だけ作ってよい（保存できる定義ファイルの数に制限はない）。その上で、プロとして次の目安を一度だけ伝える：同時に並列で動かすのは3〜5体が効率のピーク（それ以上は報告のとりまとめに時間を食う。ただし定義として持っておく分には何体あっても問題ない — 使うときに呼ばれるのは必要な部下だけ）。段階導入（まず数体→様子を見て拡大）を提案してもよいが、**最終決定は常にユーザー**。全部一括で欲しいと言われたら全部作る。

## 品質チェックリスト（生成後に必ず自己点検）

- [ ] nameは小文字とハイフンのみ・重複なし
- [ ] descriptionに「何をするか」＋「どんな場面で使うか」の両方がある
- [ ] toolsは仕事に必要な最小限か（書き換え不要ならWrite/Editなし）
- [ ] modelは仕事の難易度に見合っているか（過剰スペックにしていないか）
- [ ] 本文に出力形式・報告の形が定義されているか
- [ ] 危険な操作（削除・外部送信・本番環境）への歯止めが本文にあるか
