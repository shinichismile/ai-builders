# 検査項目チェックリスト（部品別）

問診で不調が出た部品は全項目、それ以外はざっと流す。「症状→よくある原因」の対応表も併記。

## CLAUDE.md（役割・ルールの中枢）

- [ ] 現実とズレたルールはないか（仕事のやり方が変わったのに古い手順が残っている）
- [ ] 互いに矛盾するルールはないか（後から足したルールが先のルールと衝突するのは典型症状）
- [ ] この1ヶ月で一度も役に立っていないルールはないか（→削除候補。迷ったら「このルールが無かったら何が困るか」で判定）
- [ ] 肥大化していないか（目安: 実働ルールが100行を大きく超えたら、知識部分を参照ファイルへ分割）
- [ ] 日付・数字・固有名詞が古くなっていないか

## スキル

- [ ] **発動チェック**：descriptionに実際の頼み方フレーズが入っているか。「発動しない」症状の8割はここ
- [ ] **取り合いチェック**：複数スキルのdescriptionを並べて読み、同じ依頼が2つ以上にマッチしないか
- [ ] 本文が肥大化していないか（500行超は参照ファイルへ分割）
- [ ] 「毎回同じ手直し」が放置されていないか（→その修正を本文に反映）
- [ ] 履歴・ログ系ファイル（history等）が無限に育っていないか（→月次アーカイブ方式を提案）
- [ ] 1ヶ月以上使われていないスキルはないか（→削除でなくまず理由を確認。時期物なら残す）

## サブエージェント

- [ ] **委任チェック**：親から実際に仕事を任されているか。任されないならdescriptionが曖昧（「〜のときは必ず使うこと」の一押しがあるか）
- [ ] 権限過剰はないか（読むだけの仕事にWrite/Edit、不要なBash等。最小権限に絞る）
- [ ] モデル過剰はないか（定型調査にopus等。コストに直結するので見直し価値が高い）
- [ ] 担当の重複はないか（2体が同じ依頼を取り合っていないか）
- [ ] memory持ちの場合：古い記憶で誤報告するリスクに鮮度ルール（◯ヶ月で再確認）があるか

## メモリ

- [ ] 事実が古くなっていないか（引っ越した・料金が変わった・プロジェクトが終わった等）
- [ ] 重複・矛盾するメモリはないか（統合する）
- [ ] 「その時だけの作業ログ」が紛れ込んでいないか（次回も使う事実だけ残す）

## 自動化（定期実行・フック）

- [ ] 失敗し続けているタスクはないか（動かないまま放置が一番多い）
- [ ] もう要らない定期タスクはないか（目的が終わったのに走り続けている）
- [ ] フックのシェルコマンドに安全上の問題はないか

## セキュリティ（全部品横断・毎回必ず）

- [ ] 設定ファイル・スキル・メモリにパスワード・APIキーが直書きされていないか
- [ ] 確認スキップ系の設定が有効になっていないか
- [ ] 人に渡す予定のスキルに、個人情報・顧客情報・内部ルールが混ざっていないか

## 症状→よくある原因の早見表

| 症状 | まず疑う場所 |
|---|---|
| スキルが発動しない | そのスキルのdescription（頼み方フレーズ不足） |
| 違うスキルが発動する | 複数スキルのdescriptionの取り合い |
| 部下に仕事が回らない | サブエージェントのdescription（一押し不足） |
| 応答の質が落ちた気がする | CLAUDE.mdの肥大・矛盾ルール |
| 前に言ったことを忘れる | メモリの未記録 or 散らかり |
| 動作が遅い・コストが増えた | スキル本文の肥大／モデル過剰／履歴ファイルの無限成長 |
| 同じ間違いを繰り返す | 「毎回の手直し」がスキル・ルールに未反映 |
