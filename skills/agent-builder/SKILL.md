# 自分専用エージェント・ビルダー

「自分専用エージェント」とは、高いモデルを漠然と使うことではなく、**役割 × 知識 × 道具 × 記憶 × 自動化 × 安全** の6要素を自分の仕事に合わせて設計することで生まれる。設計が良ければ、同じモデルでも体感品質はまるで変わる。逆に設計がなければ、どんなに賢いモデルも「毎回自己紹介からやり直す物知りな他人」のまま。

このスキルは、AIの専門知識がない人でも、ヒアリングに答えるだけで自分専用エージェントの設計図と実体（設定ファイル一式）が手に入るように作られている。

## エージェントを構成する6要素

1. **役割**：何者として振る舞うか（秘書・番頭・編集者・参謀…）。口調・報告スタイルも含む → CLAUDE.md に定義
2. **知識**：ユーザーの仕事・業界・用語・判断基準・よくある状況 → CLAUDE.md と参照ファイルに定義
3. **道具**：使えるツールと外部接続（ファイル・Web・メール・カレンダー等のMCP） → 接続設定と権限
4. **記憶**：会話をまたいで覚えておくこと（好み・決定事項・進行中の案件） → メモリの仕組み
5. **自動化**：毎日・毎週決まってやること（朝のまとめ、定期レポート等） → 定期実行タスク
6. **安全**：やってはいけないこと・確認を取るべきこと → 権限設計とルール

実装方法の詳細は `references/components.md` にある。**構築フェーズに入ったら必ず読むこと。**

## ワークフロー

### 1. ヒアリングする（省略しない）
本題に入る前に、まず利用環境を確認する：「Claude Code（デスクトップアプリ/CLI）を使っていますか？ それともブラウザ版Claude（claude.ai）だけですか？」。このスキルの構築手順（CLAUDE.md・スキル・hooks等のファイル配置）はClaude Code前提。ブラウザ版だけの人には、Projects機能のカスタム指示に翻案する旨を先に伝える。

`references/hearing-guide.md` の質問リストに沿って、**一度に2〜3問ずつ**聞く（アンケート用紙のように全部並べない。会話として進める）。相手の回答から次に聞くべきことを選ぶ。専門用語が通じない相手には、例え付きの聞き方（同ファイルに記載）を使う。

STEP 1〜4は**すべて聞き切る**。すでに会話の中で分かっていることだけは聞き直さない。聞き終えたらガイド内の「ヒアリング完了の確認」サマリー表で認識合わせをし、ユーザーの合意を得てから次へ進む。ヒアリングの深さがそのまま完成品の質になる。

### 2. 設計図を提示して合意を取る
6要素それぞれについて「何を・どう作るか・なぜか」を1枚の表にまとめて見せる。`references/templates.md` に完成例が3業種分あるので、近いものを下敷きにする。ここでユーザーの修正を受けてから構築に入る。

### 3. 構築する
`references/components.md` の実装ガイドに従って、必要なファイル・設定を作る：

- **CLAUDE.md**（役割・知識・ルール）— 中核。必ず作る
- **スキル**（手順が決まった繰り返し作業がある場合）— skill-builder スキルと連携して作る
- **メモリ**（記憶の置き場所と運用ルール)
- **MCP接続の提案**（必要な外部ツールがある場合。公式・信頼できる提供元のみ）
- **定期実行**（決まった時間にやることがある場合）

一度に全部作らなくてよい。**最初はCLAUDE.md＋メモリだけの最小構成で動かし始め、使いながら足すのが最短の成功ルート**（最初から全部盛ると、どの部品が効いているか分からなくなる）。

### 4. 試運転する
普段の仕事を1つ、実際に頼んでもらう。「役割どおりの口調か」「知識が反映されているか」「余計な確認・足りない確認はないか」を観察し、CLAUDE.mdに反映する。

### 5. 育てる運用を教える
エージェントは作って終わりではなく育てるもの。ユーザーに次の習慣を案内する：

- 気に入らない応答があったら、その場で「今後はこうして」と言う（→メモリ・CLAUDE.mdに反映される）
- 月に1回、ai-checkupスキルで健康診断する（古いルールの削除・メモリ整理・発動しないスキルの修理。増やすより減らす方が効く）
- 新しい繰り返し作業に気づいたら、skill-builderでスキル化を検討する

## 安全の原則（全業種共通・妥協しない）

- **扱うデータが「自分のものか、勤務先のものか」を必ず確認する**。会社員の場合、顧客情報や営業秘密を個人PCのメモリファイルに溜める設計は、勤務先のAI利用ポリシーや守秘義務に反する恐れがある。勤務先のデータを扱う設計になりそうなら「会社のルールを先に確認してください」と一言添え、確認が取れるまでは個人情報を含まない形（匿名化・件名だけ等）で設計する。ユーザーが「大丈夫」と言っても、確認が済んでいない段階では実名・取引条件を記憶に残さない設計を既定にする（緩めるのは確認後）
- APIキー・パスワード等のシークレットは設定ファイルに直書きしない（環境変数と`.gitignore`を徹底）
- 破壊的操作（削除・上書き・送信）は必ず確認を取る設計にする。確認スキップ系の設定（`--dangerously-skip-permissions`等）は勧めない
- MCPサーバーは公式・信頼できる提供元のみ。追加時は何にアクセスできるようになるかを説明する
- 医療・金融・法律のデータを扱う場合は、規制要件（GDPR/HIPAA等）への言及を怠らない
